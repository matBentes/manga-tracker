## Codebase Patterns
- Backend uses Gradle (NOT Maven despite PRD docs), Java 21, Spring Boot 3.4.4
- Flyway migrations in: backend/src/main/resources/db/migration/
- Migration naming: V{n}__{description}.sql (e.g. V1__create_manga_table.sql)
- application.properties uses env var fallbacks: ${DB_URL:jdbc:postgresql://localhost:5432/manga_tracker}
- ddl-auto=validate (JPA validates against Flyway schema at startup)
- Java/JVM is NOT installed in the devcontainer — typecheck verification is by inspection
- Docker is NOT available — Testcontainers integration tests cannot run locally
- Checkstyle config: backend/config/checkstyle/checkstyle.xml
- Scraper layer in package com.mangaTracker.backend.scraper; all scrapers registered as @Component and injected as List<MangaScraper> into ScraperRegistry
- Jsoup dependency already present in build.gradle (org.jsoup:jsoup:1.18.3); no pom.xml in this project
- SakuraMangasScraper uses multi-selector fallback pattern (try multiple CSS selectors in order); selectors may need updating if sakuramangas.org HTML changes
- Exception classes live in com.mangaTracker.backend.exception package
- JPA entities use Lombok: @Entity @Table @Getter @Setter @Builder @NoArgsConstructor @AllArgsConstructor
- Use jakarta.persistence (NOT javax.persistence) — Spring Boot 3 uses Jakarta EE 10
- Hibernate @CreationTimestamp/@UpdateTimestamp for created_at/updated_at columns
- CI runs spotlessApply (auto-formats) before checkstyle; minor formatting issues are auto-fixed and committed by CI
