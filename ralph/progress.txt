## Codebase Patterns
- Backend uses Gradle (NOT Maven despite PRD docs), Java 21, Spring Boot 3.4.4
- Flyway migrations in: backend/src/main/resources/db/migration/
- Migration naming: V{n}__{description}.sql (e.g. V1__create_manga_table.sql)
- application.properties uses env var fallbacks: ${DB_URL:jdbc:postgresql://localhost:5432/manga_tracker}
- ddl-auto=validate (JPA validates against Flyway schema at startup)
- Java/JVM is NOT installed in the devcontainer — typecheck verification is by inspection
- Docker is NOT available — Testcontainers integration tests cannot run locally
- Checkstyle config: backend/config/checkstyle/checkstyle.xml

## 2026-03-01 - US-002
- What was implemented: Created V2__create_notification_log_table.sql and V3__create_app_settings_table.sql Flyway migrations

- Files changed:
  - backend/src/main/resources/db/migration/V2__create_notification_log_table.sql (created)
  - backend/src/main/resources/db/migration/V3__create_app_settings_table.sql (created)
  - ralph/prd.json (US-002 passes set to true)
- **Learnings for future iterations:**
  - app_settings uses INTEGER PK (not UUID) because it's a single-row config table — id is always 1
  - The ON DELETE CASCADE on notification_log.manga_id FK means deleting a manga automatically cleans up its notification logs (important for MangaService.deleteManga)
  - app_settings seed row is inserted in the V3 migration itself; no separate seed migration needed
  - Both UNIQUE constraint and UNIQUE INDEX created (same pattern as V1 for consistency)
