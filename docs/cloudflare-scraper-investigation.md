# The Cloudflare Wall — Why the Sakura Scraper Can't Fully Be Automated

A write-up of investigating why `sakuramangas.org` scraping fails, what the site's
protection actually is, and the honest engineering decision that came out of it. Written as a
companion to `aws-deploy-journey.md` — the Cloudflare story it promised as "probably a
separate post."

> **TL;DR:** The site is behind Cloudflare Bot Management with an interactive "Verify you are
> human" challenge. You can't beat that from a datacenter (AWS) or an unattended bot — but a
> real browser on a trusted home IP clears it every day. So the project's endgame is: run the
> scraper **locally** from a trusted IP, make it **fail gracefully** when blocked, and accept
> that fully-unattended cloud scraping of this particular site isn't achievable without
> crossing lines that aren't worth crossing.

## The symptom

Adding a manga from `sakuramangas.org` sometimes works, sometimes fails with the page stuck on
"Just a moment…" / "Verify you are human." Critically, **a human visiting the site in a normal
browser also sees the checkbox** — so this isn't a bug in our scraper; it's the site's
enforcement level going up.

The scraper (`SakuraMangasScraper` + `PlaywrightBrowserManager`) already uses a real headless
Chromium with anti-fingerprinting stealth flags. That was enough for the *old*,
auto-solving Cloudflare challenge. It is not enough for the *interactive* one.

## "Just use their public API" — a dead end, with evidence

The manga are viewable without logging in, so the obvious idea is: skip the browser, call the
site's internal JSON API directly. We captured a real browser session as a HAR file and
looked at exactly what those API calls carry. Two endpoints serve the data:

- `…/manga/..__obf__manga_info.php` → returns cleartext JSON (title, author, synopsis)
- `…/manga/..__obf__manga_capitulos.php` → returns the chapter list

Both succeeded from a real browser. But each request carried a set of tokens that make direct
replication impractical — **three locks**, and "no login required" defeats none of them:

**Lock 1 — the tokens are generated by obfuscated client-side JS.** Every request sends
`x-csrf-token`, `x-client-signature`, `x-verification-key-1`, `x-verification-key-2`, a
`proof`, and a `challenge`. These are computed fresh by the site's own obfuscated security
script. Sending them from a plain HTTP client means reverse-engineering and reimplementing
that script — which the operator can re-obfuscate at will.

**Lock 2 — the token is bound to the client's IP.** The `challenge` parameter base64-decodes
to `<client-IPv6>/<hash>/<hash>` — the requester's own IP address is baked into the token. A
token minted in a home browser is rejected when replayed from a different IP (e.g. an AWS
box). This is the same IP-binding wall as Cloudflare's own clearance cookie: you cannot
"solve it here, use it there."

**Lock 3 — the chapter payload is encrypted.** The chapters response isn't readable JSON —
it's `{"_enc":"…"}`, ciphertext only the site's JS knows how to decrypt.

So calling the API directly would mean: reimplement the token generator, spoof the bound IP,
*and* reverse the payload decryption — against a moving target. That is strictly harder than
the current approach, which is exactly why the original design abandoned direct API calls and
switched to "let a real browser run the site's JS, then read the rendered chapter list from
the DOM." Letting the browser run the JS outsources all three locks to the site's own code.

## The real insight: it's about IP reputation, not code

The same scraper code that fails on AWS **works from a home machine.** The reason is how
Cloudflare picks a challenge level per request, driven mostly by IP reputation:

- **Trusted IP (residential home broadband)** → Cloudflare serves the invisible, self-solving
  JS challenge, or waves you through. Our stealth Chromium *can* clear that. → scrape succeeds.
- **Low-trust IP (AWS/datacenter range)** → Cloudflare escalates to the interactive checkbox.
  No amount of fingerprint stealth beats it, because the block is on the IP, not the browser.
  → scrape fails.

This matches the code's own comment: *"Cloudflare Bot Management blocks scrapes from the
Fargate IP range regardless of fingerprint stealth (IP reputation)."* Same binary, different
IP, opposite outcome. The local success isn't luck — it's the trusted IP earning the beatable
challenge.

## Why not a CAPTCHA-solving service?

The tempting "just automate the click" fix is a paid CAPTCHA-solving service. We chose not to,
on principle, not just pragmatism:

- The checkbox exists to answer *"is a real human here?"* When **you** click it, the honest
  answer is yes — you're satisfying the check. A solving service makes it pass when **no human
  is present** — that's *circumventing* the access control, which is the single clearest
  violation of the "don't bypass our security measures" clause essentially every site's terms
  carry. Deploying an interactive challenge is an explicit "no unattended bots" signal.
- Practically, it's also a paid, brittle dependency in a fight the operator keeps re-tuning.

The occasional-manual-click approach avoids all of that: it's honest about a human being
present, it's free, and it's stable.

## Why this doesn't become a multi-user product (as-is)

A natural next thought is "make it collaborative — users solve the captcha to keep it
working." That breaks on the same IP-binding: a clearance solved by one user on their IP is
useless to a central server or to another user. The only technically-coherent multi-user shape
is **per-user client-side scraping** (a browser extension where each user's own browser scrapes
their own titles) — a much larger product, and one that still raises redistribution/terms
questions once a server re-serves one user's scraped data to others. For a shared product, the
right backbone is an API-first source that *permits* programmatic access; Sakura can only be an
optional, per-user, bring-your-own-session add-on.

## The decision

For a personal pt-BR manga notifier, you can't have all three of: **(1) Sakura as the source,
(2) fully unattended, (3) running from the cloud.** Pick which to drop. We drop **cloud**:

1. **Run the poller from a trusted (home) IP**, where the scraper already works, with a
   persistent browser profile so the Cloudflare clearance cookie survives between polls — one
   manual checkbox solve then covers hours-to-a-day of hands-off polling across the whole list.
2. **Make the scraper degrade gracefully** when it *is* blocked: detect the challenge page and
   surface a clear "source temporarily unavailable / blocked by Cloudflare" state instead of a
   raw error or a silent no-op, so a block is visible rather than mysterious.
3. **Keep the AWS deployment** as the showcase of the full stack — just don't ask it to also
   win the Cloudflare fight from a datacenter IP.

## Lesson for future-me

Anti-bot walls aren't a bug to fix; they're a decision by the site owner, working as intended.
The engineering win here wasn't beating Cloudflare — it was *measuring the wall precisely*
(the HAR, the three locks, the IP-reputation behavior), then choosing the honest architecture
that fits it, and making the software fail cleanly at the boundary instead of pretending the
boundary isn't there.
